## 1. 서비스 한 줄 정의

**"기존 소개팅 앱은 지인을 '피하게' 해주지만, 이 서비스는 지인 관계를 '드러내는' 것이 셀링 포인트다."**

- 신뢰 기반: 회사 이메일 인증 + 지인 추천코드 필수 → 모든 회원이 "누군가의 지인"
- 핵심 재미: 프로필마다 나와의 관계 경로 표시 — "짱구님의 지인(영희)의 지인(맹구)"
- 중매인 모드: 유희왕 카드덱처럼 내 지인들을 등록·관리하고 직접 매칭 제안

## 2. 타겟 & 규모

- 1차 타겟: 운영자 본인의 지인, 직장동료 및 그들의 지인 네트워크 (수십~수백 명)
- 연령대: 20대 후반 ~ 40대 초반 직장인
- 성격: 소소한 재미 + 실제 매칭. 상용 서비스가 아닌 프라이빗 클럽 성격 (Raya 참고)

## 3. 플랫폼 결정

- **모바일 웹 (반응형, 모바일 퍼스트)** 로 개발. 앱스토어 미출시.
- 이유: (1) 카톡 링크 공유가 핵심 유입 경로 (2) 데이팅 앱 스토어 심사 회피 (3) 개발 속도
- Phase 3에서 PWA 전환 (홈 화면 추가 + Web Push 알림)

## 5. 회원 유형

| 유형 | 설명 | 필수 입력 |
|---|---|---|
| 중매인(Matchmaker) | 지인 추천 목적. 본인은 소개팅 안 함 | 이름, 연락처, 생년월일, 직장 |
| 솔로(Single) | 본인 소개팅 목적 | 중매인 항목 + 소개팅 프로필 전체 |

## 6. 가입 & 인증 정책

### 6.1 가입 절차 (2단계만)
1. **회사 이메일 인증** — 이메일로 6자리 코드 발송, 입력 시 인증 완료 (블라인드 방식)
2. **지인 추천코드 입력 (필수)** — 기존 회원의 코드. "어떻게 알고 오셨나요?" 대체. 코드 입력 시 추천인과의 **관계 유형** 선택: 지인 / 가족 / 직장동료 / 선배 / 후배 / 사촌 / 기타
3. 이후 기본 정보(이름, 연락처, 생년월일, 직장) 입력 → 가입 완료

### 6.2 회사 이메일이 없는 경우 (예술인, 프리랜서 등)
- **지인 보증(Vouch)** 방식: 이미 인증된 회원 1명이 앱 내에서 "보증하기" 승인 → 가입 허용
- 보증인은 추천인과 동일인이어도 무방 (추천코드 발급자에게 보증 요청 알림 발송)
- 서류 인증(재직증명서 등)은 운영 부담이 커서 도입하지 않음. 골드스푼식 서류 심사는 규모가 커지면 검토.

### 6.3 개인정보 동의 (가입 시 필수 체크)
- [필수] 서비스 이용약관 / 개인정보 수집·이용 동의
- [필수] **"내 이름이 다른 회원의 관계 경로에 표시되는 것에 동의합니다"** — 미동의 시 경로에서 "지인"으로 익명 처리
- [필수] 매칭 성사 시 전화번호 상대방 공개 동의 (프로필 등록자 기준)
- 지인 프로필 대리 등록 시: **당사자 동의 확인 체크박스 필수** + 당사자 연락처로 등록 사실 SMS/알림 발송 (분쟁 방지 핵심)

### 7.2 사진 정책 (v1.5 대칭 개정)
- **성별 무관 전면 비공개**: 사진은 본인·등록자와 사진 교환(EXCHANGE)/성사(MATCH) 열람권 보유자에게만 (§9.0 v1.5)
- 권한 없는 카드·상세는 **카드 뒷면/실루엣 처리**. CSS blur 금지 — 원본 URL이 노출되므로, 권한 없는 요청에는 원본을 아예 응답하지 않는다 (서버 사이드 gate)
- 열람자에게는 등록된 개수만큼 그대로 노출 (등록 개수 = 노출 개수)
- 업로드 시 리사이징(장변 1600px) + WebP 변환, EXIF(위치정보) 제거
- 우클릭/롱프레스 저장 방지 + 열람자 닉네임 워터마크 (Phase 2, 유출 방지)
- **중매인도 성별 규칙 동일 적용**: 남성 중매인은 여성 사진 열람 불가 (본인이 직접 등록한 자기 덱의 여성만 예외). 중매인 계정이 사진 열람 우회로가 되는 것을 차단
- **대리 등록 여성의 열람 수락 주체**: 등록 시 "사진 열람 수락 권한을 중매인에게 위임" 별도 동의 체크(`delegatePhotoConsent`). 위임이 없으면 중매인은 당사자 확인 후 응답 입력을 원칙으로 함

### 7.3 노출 회피 (v1.1 추가)
- **같은 회사 상호 비노출**: 프로필 설정 `avoidSameCompany` (기본 ON). 열람자(User.company)와 프로필(Profile.company)의 정규화된 회사명이 같고 **어느 한쪽이라도 ON이면** 피드·상세·호감 전부에서 상호 숨김. 회사명 정규화: 소문자화 + 공백 제거 + "(주)/주식회사/㈜" 제거
- **아는 사람 피하기 (연락처 차단)**: MY → 아는 사람 피하기에서 휴대폰 번호 등록 (최대 200개). 원본 번호는 저장하지 않고 **HMAC-SHA256 해시 + 마스킹 라벨(010-****-1234)만 보관**. 등록 번호와 phoneHash가 일치하는 회원/프로필과 **상호 비노출** (내가 아는 사람을 안 보고, 그 사람도 나를 못 봄). 대리 등록 프로필 당사자는 비회원이라 차단 목록을 가질 수 없음 (Phase 2 검토)
- **특정 회사 피하기 (v1.4 추가)**: 회사명 등록 (최대 50개, `CompanyBlock` — 정규화명+원문 라벨). 정규화된 회사명이 일치하는 프로필을 숨기고, **내 회사를 등록한 회원에게도 내가 숨겨짐** (상호 비노출). 전 직장·거래처 회피 용도
- 적용 지점: 탐색 피드 제외 + 프로필 상세 404 + 호감 전송 차단 (서버 사이드)

### 7.4 프로필 클레임 — 대리 등록 당사자의 직접 가입 연동 (v1.2 추가)
- **연동 조건**: 가입 시 입력한 **이름 + 연락처(해시)가 정확히 일치**하는 미연동 대리 등록 프로필(`isSelf=false, userId=null`)이 있으면 새 계정에 자동 연동 (`userId` + `claimedAt` 세팅). 일치 프로필이 여럿이면 가장 먼저 등록된 것 하나만 연동
- **편집 권한**: 연동 후 당사자는 항상 편집 가능. 추천인(등록자)의 편집 권한은 당사자가 결정:
  - 온보딩(또는 MY > 내 프로필)에서 질문 → **같이 수정**(`ownerCanEdit=true`) 또는 **나만 수정**(`ownerCanEdit=false`, 추천인은 열람만)
  - 결정 전까지는 추천인 편집 가능 (기존 동작 유지). 결정 시각은 `editShareDecidedAt`
- **알림**: 연동 시 추천인에게 `PROFILE_CLAIMED`, 권한 결정 시 `EDIT_SHARE_DECIDED`
- 연동된 프로필은 당사자의 "내 프로필"로 동작 (호감 전송·사진 gate·회피 설정 등). 연동된 계정은 별도 본인 프로필을 새로 만들 수 없음 (중복 방지)

### 7.5 프로필 수정 (v1.2 추가)
- 모든 항목 수정 가능: `/p/[id]/edit` (다단계 폼 재사용, 사진 관리 포함)
- 권한: 당사자(본인/클레임) 또는 편집 권한 있는 등록자 (`canEditProfile`)
- 자유 텍스트 항목은 생성과 동일한 연락처 우회 감지 적용
- **이름·연락처는 최초 등록 후 수정 불가** (v1.3): 타인 명의 도용·바꿔치기 방지. 수정 API 스키마에서 제외 + 폼에서 잠금 표시. 오기입 시 관리자 문의로 처리 (연락처 원본은 수정 화면에도 마스킹 전달)

### 7.6 랜덤 닉네임 — 성사 전 실명 비공개 (v1.3 추가)
- 프로필 생성 시 랜덤 닉네임 부여 (`Profile.nickname`, 예: "포근한 수달") — 이후 고정
- **실명 공개 조건**: ① 내 프로필(당사자·등록자) ② **성사(Match)된 상대**. 그 외 모든 화면·API는 닉네임만 노출 (탐색 피드, 프로필 상세, 호감함, 관계 경로 nodes, GET /api/profiles/[id])
- 연락처 공개 조건(성사 시)과 동일 기준으로 통일 — "서로 호감이 확정됐을 때만 신원 공개"
- 관계 경로의 **중간 인물(회원) 실명**은 기존대로 `allowNameInPath` 동의 기반 (§8.2)

### 7.7 비가입자 공유 링크 (v1.3 추가)
- 편집 가능자(등록자/당사자)가 프로필 상세에서 "📤 공유" → `Profile.shareToken`(랜덤 32hex) 발급 → `/s/[token]` **공개 페이지** (로그인 불필요). 모바일에선 Web Share 시트(카카오톡 포함), 데스크톱은 링크 복사
- **공개 범위**: 닉네임·나이·지역·직무·키·체형·라이프스타일·MBTI·취미·이상형·한 줄 소개. **실명·연락처·회사명 미포함**. 사진은 남성만 공개, 여성은 잠금 안내 (§9.0 동일 기준)
- ACTIVE 프로필만 열람 가능 (숨김/성사 시 "지금은 볼 수 없는 프로필" 안내)
- 하단 CTA "다리 가입하기" → `/signup?code={등록자 추천코드}` 로 추천코드 자동 입력 (바이럴 경로)
- 회수: PATCH `action:"unshare"` 로 토큰 무효화 (UI는 TODO(phase-1.5))

### 7.9 인적사항 미상 등록 (v1.4 추가, v1.5 전 관계 확대)
- **모든 대리 등록 관계**에서 당사자 이름·연락처를 모르면 **호칭(`pendingLabel`, 예: "회사 후배")으로 등록** (`identityPending=true`). 당사자 name은 빈 값, phoneHash는 null. 이름·연락처는 나중에 등록자가 직접 입력하거나(상세 → ✍️ 입력하기), 다리 지인 정보(`via*`, 선택 입력)를 적어두면 그 지인이 가입 시 안내를 받아 입력
- **미입력 상태 제한**: 탐색 피드·상세(등록자 외)·호감 수신·공유 링크 전부 차단. 등록자 카드덱에는 "⏳ 지인(OO)의 정보 입력 대기" 배지
- **다리 지인이 가입하면** (이름+연락처 해시가 via*와 일치): 알림 `PROFILE_IDENTITY_NEEDED` + 홈 상단 배너 → `/me/identify` 목록 → `/p/[id]/identify`에서 당사자 이름·연락처 입력 + **당사자 동의 확인** → `identityPending` 해제, 정상 공개. 등록자에게 `PROFILE_IDENTITY_FILLED` 알림
- 입력 권한: 다리 역할 지인(매칭 회원) 또는 등록자. 입력은 "최초 설정"이므로 §7.5 불변 규칙과 충돌 없음 (입력 후엔 동일하게 잠김)
- 입력 시 당사자가 이미 회원이면 즉시 클레임 연동 (§7.4)

### 7.10 회원 탈퇴 (v1.4 추가)
- MY → 회원 탈퇴 → 안내 확인 + "탈퇴합니다" 입력 → `DELETE /api/me`
- **개인정보 익명화 방식** (Like/Match/추천 그래프 FK 무결성 유지를 위해 행은 보존):
  - User: email 무효화(재로그인 차단), 이름 "탈퇴 회원", 전화·회사 삭제, `deletedAt` 기록, 경로 이름 비노출
  - 내 프로필 + 내가 등록한 미클레임 대리 프로필: 사진(Storage 포함)·이름·연락처·소개글 삭제, 영구 HIDDEN, 공유 토큰 무효화. **타 회원에게 클레임된 프로필은 당사자 소유이므로 유지**
  - 진행 중 호감 EXPIRED 처리, 사진 열람권 전부 회수, 차단 목록·알림·인증코드 삭제
- 잔여 JWT 세션은 `requireUserId`(전 API)와 (main) 레이아웃에서 `deletedAt` 확인으로 차단
- 이미 성사되어 공개된 전화번호는 되돌릴 수 없음을 탈퇴 화면에 고지

### 7.8 중매인 모드 열람 (v1.3 명확화)
- **본인 프로필 없이도 탐색 피드·프로필 상세 열람과 지인 추천은 자유** (호감 전송만 본인 프로필+사진 필요)
- 홈 배너: 카드덱만 있는 회원에겐 "중매인 모드" 안내로 대체 (프로필 강요 문구 제거)
- 남성 중매인의 여성 사진 열람 불가 원칙(§7.2)은 유지

## 9. 매칭 플로우

### 9.0 대칭 사진 교환 정책 (v1.5 전면 개정 — 핵심 원칙)
> (구) 비대칭 정책(여성만 사진 gate)은 v1.5에서 폐기. **성별 무관, 모든 사진은 기본 비공개**로 강화.
- 탐색 피드는 **카드덱 형태**: 모든 프로필이 카드 뒷면(🃏)으로 놓이고, 탭하면 카드를 뽑듯 뒤집혀 **텍스트 프로필만** 나타난다 (사진 없음). 뒷면 색상으로 성별 표시 — **남성 파랑 계열 / 여성 분홍 계열**
- 사진 미등록 프로필도 활동 가능 (호감 전송 포함) — 카드·상세에 "📷 사진 미등록" 표시로 상대가 미리 알 수 있게
- 사진은 **서로 사진 교환을 수락한 두 사람 사이에서만** 상호 공개 (`PhotoAccess`, source=EXCHANGE — 양방향 동시 발급)
- 연락처는 그다음 단계 — 사진까지 확인하고 최종 수락(성사)해야 공개
- 정책 슬로건: **"결정은 당당하게, 대신 빠르게"** — 거절은 양쪽 모두의 정당한 권리이되, 상대를 기다리게 하지 않는다.

### 9.1 직접 호감 (성별 무관 대칭 3단계, v1.5)
```
1단계  텍스트 프로필·관계 경로만 보고 [💚 호감 보내기]
   ↓  상대에게 알림 (양쪽 다 사진은 아직 비공개)
2단계  상대가 [📸 사진 교환 수락] → 상태 PHOTO_GRANTED → 두 사람의 사진이 서로에게만 동시 공개
       상대가 거절 → 사유 프리셋 전달, 상호 영구 비노출 (종료)
   ↓
3단계  발신자가 사진 확인 후 결정 (⏳ 48시간 기한)
   ├─ [📞 연락처 교환] → 매칭 성사 → 양측 전화번호 상호 공개
   ├─ 거절     → 사유 프리셋 전달 → 상호 영구 비노출
   └─ 무응답   → 자동 만료. 상대에게는 "기한 만료"로만 표시 (사진 열람 여부는 비공개)
```

**공통 규칙**
- **거절 사유는 자유 서술 금지, 프리셋 칩만**: ① 지금은 연애 생각이 없어요 ② 이상형과 조금 달랐어요 ③ 아는 사이라서요 ④ 거리가 멀어서요 ⑤ (사유 없이 거절) — **외모 관련 사유는 프리셋에 없음** (사진 공개 직후 거절이라도 사유가 외모로 특정되지 않도록)
- 호감 발신 제한: 1인당 동시 진행 3건 (스팸 방지 + 여성 수신함 보호)
- 1단계 응답 기한: 7일 무응답 시 자동 만료 / 3단계(사진 열람 후) 결정 기한: **48시간**

**잠수 방지 완충장치 (사진 열람 후 무응답 대응)**
- 3단계 무응답 만료가 30일 내 2회 누적 시, 동시 진행 한도 3건 → 1건으로 축소 (30일간)
- 거절·만료·차단 즉시 사진 접근권 회수 (`PhotoAccess` revoke — DB_SCHEMA §4.5)
- 여성 화면에는 "열람했는지 여부"를 절대 표시하지 않음 (읽씹 확인은 상처만 남김)

### 9.2 중매인 매칭 제안
```
[중매인이 내 카드덱에서 지인 A 선택] + [탐색에서 상대 B 선택] → "이 둘 어때?" 제안
   ↓
A, B 각각에게 (또는 각자의 중매인에게) 제안 알림
   ↓
양측 모두 수락 → 매칭 성사 → 번호 공개
한쪽이라도 거절 → 제안 종료 (사유 프리셋 동일)
```
- 비회원 지인(대리 등록된 프로필)의 수락/거절은 담당 중매인이 대리 응답 (카톡으로 물어보고 입력하는 기존 방식의 디지털화)

### 9.3 매칭 성사 후
- 양측 전화번호 공개 + "성사 축하" 화면
- 해당 프로필은 상태가 `MATCHED(졸업)`로 변경 → 탐색 피드에서 자동 숨김
- 중매인 개입 건이면 중매인 성사 카운트 +1

## 10. 카드덱 (중매인 모드)

- 내가 등록한 지인 프로필들이 **카드 그리드**로 표시 (대표사진 + 이름 + 나이 + 한 줄 소개)
- 카드 상태 컬러:
  - ⬛ 검정 테두리 = 대기 (매칭 진행 없음)
  - 🟨 노랑 테두리 = 호감/제안 진행 중
  - 🟩 초록 테두리 = 매칭 성사 (졸업/탈출)
- 카드 탭 → 지인 프로필 관리 (수정 / 숨김 / 삭제)
- 카드 길게 누르기 or "매칭 제안" 버튼 → 상대 탐색으로 이동해 제안 시작

## 11. 게이미피케이션

- 뱃지: 첫 성사 🥉 / 3건 🥈 / 5건 🥇 / "가문의 영광"(가족 매칭 성사) 등
- 촌수 뱃지: 성사 커플의 관계 거리 표시 ("4다리 건너 만난 커플")

## 12. 안전: 신고 · 자동 감지 · 제재 · 관리자

### 12.1 신고 (v1.4에서 프로필 단위 구현)
- 신고 사유 7종: **음란성 / 광고·스팸 / 불법(도박·성매매 등) / 허위 정보 / 당사자 미동의 등록 / 연락처·SNS 노출 / 기타**
- 신고 대상: 프로필 전체 또는 개별 사진·텍스트 항목 (개별 항목 신고는 Phase 2). 접수 즉시 관리자 큐 적재, 24시간 내 처리 목표
- 동일 대상 신고 3건 누적 시 자동 임시 숨김 (선조치 후검토). 관리자 기각 시 남은 미처리 신고가 없으면 자동 재공개, 확정 시 영구 숨김
- 차단: 특정 회원/프로필 상호 비노출 — 신고와 별개, 사유 불요, 즉시 적용

### 12.2 자동 감지 파이프라인
| 대상 | 방식 | 시점 | 조치 |
|---|---|---|---|
| 텍스트 (자기소개·이상형·연애관·한 줄 소개) | 금칙어 사전 + 정규식 (URL, 전화번호, 카톡·인스타 ID, 계좌번호, 도박·성매매 키워드) | 저장 시 동기 검사 | 확정 패턴 → 저장 거부(사유 안내) / 의심 수준 → 게시 유지 + 검토 큐 |
| 사진 | 이미지 모더레이션 API (Google Vision SafeSearch / AWS Rekognition / Sightengine 중 택1) | 업로드 직후 비동기 | 음란성 고점수(≥0.9) → 자동 비공개 + 큐 / 중간 점수 → 공개 유지 + 큐 |
| 행동 | 레이트 리밋 (가입 직후 대량 호감 발신, 단시간 반복 신고 등) | 실시간 | 임계 초과 시 액션 차단 + 플래그 |

- **연락처 우회 교환 차단이 특히 중요**: 성사 전 번호·SNS 교환은 거절/차단/열람권 회수 안전장치를 전부 무력화하므로 자유 텍스트 필드는 상시 감지
- 프리셋 입력(거절 사유, 취미 칩 등)은 감지 대상에서 제외 — 자유 텍스트만 검사

### 12.3 제재 (3단계)
경고 → 7일 정지 → 영구 차단. 단, 음란물·불법 콘텐츠는 1회에도 즉시 차단 가능. 모든 제재는 사유·근거 플래그·처리 관리자와 함께 Sanction 테이블에 기록 (감사 추적)

### 12.4 관리자 대시보드 (ADMIN 전용, v1.4에서 1차 구현 — /admin)
- 접근: `role=ADMIN` + middleware gate. 일반 회원 UI에는 진입점 자체를 렌더링하지 않음
- 핵심 지표: 회원(총원·주간 신규·**성비**·역할 분포) / 프로필(활성 수·대리 등록 비율) / **매칭 퍼널**(호감 발신 → 사진 공개[여성 수락률] → 최종 성사[남성 확정률]) / 성사(누적·월간·중매 vs 직접 비율·평균 촌수) / 안전(미처리 신고·감지 대기·제재 현황)
- 성비는 데이팅 서비스 1순위 리스크 지표 → 대시보드 최상단 상시 노출
- 화면 구성 상세: PAGE_IA §2.12 / 데이터 모델·통계 쿼리: DB_SCHEMA §6·§7
